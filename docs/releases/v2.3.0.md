# v2.3.0 — Fix audio : refresh automatique des previews expirées

## Contexte

Les previews audio des playlists officielles (Années 2000, Années 1980-1990, Chanson Française, Disney, etc.) étaient des URLs Deezer signées avec un token d'expiration (`hdnea=exp=...`). Ces tokens ont expiré autour du 22 avril 2026, rendant silencieuses toutes les playlists concernées.

## Ce qui a changé

### Fix immédiat
- `buildTrack` nullifie désormais toute `preview_url` dont le token Deezer est expiré. Le jeu retombe alors sur son fallback existant (recherche Deezer + iTunes) exactement comme pour les playlists sans preview.

### Refresh automatique au démarrage
- Nouvelle colonne `preview_expires_at` sur `custom_playlist_tracks` (migration Supabase), peuplée à partir des tokens existants.
- Au chargement de chaque playlist, les tracks dont l'expiry est dans moins de 7 jours sont rafraîchies automatiquement :
  1. Tentative via `GET api.deezer.com/track/{id}`
  2. Fallback sur une recherche iTunes (URLs permanentes, sans expiry)
- Les URLs iTunes sont stockées avec `preview_expires_at = 2099-01-01` pour ne pas être re-fetchées inutilement.
- Les mises à jour BDD utilisent le client admin (service role) pour contourner les RLS.

### Résultat
Au fur et à mesure des redémarrages serveur, les URLs expirées sont remplacées par des URLs iTunes permanentes en BDD. Le fallback live reste en place comme filet de sécurité.

## Fichiers modifiés

- `src/lib/server/services/deezer.js` — `parseExpFromUrl`, `fetchDeezerTrackPreview`, `iTunesPreviewSearch`
- `src/lib/server/services/playlist.js` — `refreshExpiredPreviews`, `buildTrack` (nullification expiry)
- `src/lib/server/socket/salon.js` — appel `refreshExpiredPreviews` sur le chargement salon
- Migration SQL : `ALTER TABLE custom_playlist_tracks ADD COLUMN preview_expires_at timestamptz`
